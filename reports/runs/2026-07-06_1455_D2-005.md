## Result

Doctrine profiles, analysis findings, and reusable concepts produced by dry runs
now have a persistent, searchable home. Canonical content is git-tracked
markdown with validated front matter in `knowledge/topics/`; the derived index
(`var/topics.sqlite3`, gitignored) combines SQLite FTS5 keyword ranking with
local static embeddings (model2vec `potion-base-8M`, 256-dim, fully local — no
external API, no text leaves the machine), fused by reciprocal rank fusion.
Without the embedding stack, search degrades deterministically to keyword-only
and labels itself as such — the same architecture D2-003 requires, so its
design carries forward.

## Evidence and limitations

- Evidence: `knowledge/topics/`, `scripts/topics.py`, `tests/test_topics_store.py`.
- Limitations: potion-base-8M is a small static-embedding model — adequate at
  this corpus size, unbenchmarked beyond it; the first `index`/`search` run
  needs one network fetch to cache the model (~30 MB), after which everything
  is offline. Topic content is unreviewed working knowledge; the store's
  README forbids treating it as card provenance.
- New risks or decisions: none beyond the recorded scope insertion. The RRF
  fusion constant (60) and chunking-by-section are implementation choices, not
  frozen contract.
